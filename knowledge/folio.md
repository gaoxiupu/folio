## tl;dr

个人 Portfolio 早就不只是一个展示页面了。对产品经理、设计师和开发者来说，它更像一套对外表达自己的系统，里面放着项目经历、能力结构、思考方式，以及某种程度上的个人品牌。

传统 Portfolio 有个很实际的问题：内容越来越多，理解却没有变得更容易。访问者通常要在项目页、个人介绍、简历、文章和联系方式之间来回切换，才能慢慢拼出一个基本判断：这个人是谁，做过什么，适不适合进一步聊。

所以我做了 Folio。它想解决的，不是“个人网站要不要加一个 AI 功能”，而是另一个更具体的问题：当一个人的信息越来越丰富时，能不能用一种更自然的方式，帮别人更快理解你。

## 不是信息不够，而是理解路径太长
大多数 Portfolio 到今天还是页面导向的。作者会按照自己的逻辑整理内容，比如 About、Projects、Writing、Resume、Contact；访问者则只能顺着这些页面去翻。

但真实的访问需求并不是按页面发生的，而是按问题发生的。

招聘方通常只关心几件事：你做过什么，能力边界在哪，和岗位匹不匹配。潜在合作方会更关心你的项目经验、判断方式和合作风格。同行来看，往往是想知道你最近在做什么，怎么思考，怎么做选择。

这些问题都很具体，可传统 Portfolio 的入口不是按问题设计的。访问者得自己翻、自己找、自己拼。内容是完整的，理解路径却很长。这也是我做 Folio 的出发点，把 Portfolio 从一个等人浏览的信息集合，改成一个能直接回应问题的入口。

## 把 Portfolio 做成一个可对话的信息产品

![fangshiduibi](https://pub-fb5052d887fa4893ad36bba002744406.r2.dev/blog/fangshiduibi.jpg)

我对 Folio 的定义，不是“一个聊天插件”，而是“面向个人 Portfolio 的 AI 信息入口”。

这个定义背后有三件我比较在意的事。

第一，是降低理解门槛。访问者不需要先研究网站结构，也不需要预判信息可能藏在哪一页。他们可以直接提问，比如“你做过哪些项目”“你更擅长哪些方向”“怎么联系你”。

第二，是降低维护成本。Portfolio 主人不需要额外维护一套后台，只要把已有资料放进知识目录里，比如简历、项目文档、文章、技能信息和社交资料，系统就能基于这些内容组织回答。

第三，是保留个人表达。Portfolio 不是企业客服系统。它不只是回答事实，还要尽量保留这个人的表达方式和气质。

所以 Folio 的产品链路其实很简单：用户把资料放进知识目录，系统先整理内容，再在访问者提问时调取相关知识，最后通过聊天组件和结构化卡片把信息返回出来。整个产品以一个可嵌入的 widget 接到个人网站上。

从功能上看，它并不复杂。但这些能力都在服务同一件事：让访问者更快形成对 portfolio 主人的整体认知。

## 产品设计
### 为什么没有继续用传统 RAG

![workflow-folio](https://pub-fb5052d887fa4893ad36bba002744406.r2.dev/blog/workflow-folio.jpg)

做知识问答时，最自然的做法就是把文档切成片段，提问时做检索，再把相关内容交给模型生成答案。很多场景里，这套方法都很好用。但 Portfolio 不太一样，项目经历、职业路径、能力结构、文章观点和表达方式，本来就是连在一起的。如果太早把这些内容切成 chunk，检索当然还能工作，但这些信息之间最有价值的关系会变弱。

而对 Portfolio 来说，这些关系恰恰最重要。

访问者通常不是只想拿到一个零散答案。他们更关心的是，一个项目如何体现某种能力，一段经历如何解释职业方向，一种表达方式又如何反映这个人的判断。对这个场景来说，片段相关性不够，系统最好先有一层整体认知。

所以后来我把方案从传统 RAG 改成了“Wiki + 渐进式披露”。

Folio 不直接拿原始素材去做问答，而是先把简历、文章、PDF、Markdown、JSON 这些内容整理成结构化 wiki，按概览、经历、项目、技能这样的主题组织起来。这样，模型在回答问题时面对的不是一堆零散原文，而是一层已经整理过的知识结构。

在这个基础上，我又加了渐进式披露的策略。内容不多的时候，系统可以整体加载，优先保证回答的完整性；内容变多以后，再优先加载索引和重点页面，在上下文成本和覆盖范围之间做平衡。

### 回答不能只“成立”，还要“好读”
我在 Folio 里做了一层结构化呈现。

项目、经历、技能、联系方式这些高频信息，我都尽量设计成更适合快速阅读的形式。比如项目可以直接看到名称、简介；经历可以按公司、角色和时间段展开；技能可以用更直观的方式呈现；联系方式和 GitHub 信息也可以在合适的时候直接展示出来

### 让 AI 像你一样表达
Portfolio 从来不只是资料集合，它也在表达“这个人是什么样的人”。同样一段经历，不同的人会讲出完全不同的感觉。有人强调结果，有人强调方法，有人强调取舍，有人强调审美。访问者最终记住的，也往往不只是信息本身，还包括这种表达方式。

所以 Folio 支持 persona 设定。Portfolio 主人可以通过 persona 文件定义 bot 的语气、风格和边界，让回答更接近自己的表达方式，而不是一个标准化的话术模板。

### 边界要克制，使用门槛才会低
Folio 没有后台管理系统，没有数据库，也没有一套复杂的内容维护逻辑。整个产品是 file-driven 的：用户把已有素材放进知识目录，系统负责整理、编译和调用，最后通过一个脚本嵌到网站里。
